EasyTrip SaaS

Itinerari AI strutturati per weekend e viaggi brevi in Europa: generazione asincrona, versioni multiple, gruppi, spese condivise e monetizzazione nativa via Stripe — come implementato nel codebase easytrip-saas.

Interfaccia reale

Catture effettuate con Playwright sul dev server locale (stesso codice del repository): homepage pubblica e schermata di accesso Clerk quando si richiede /app/trips. Con E2E_AUTH_STORAGE_STATE (file JSON di sessione dopo login) si aggiungono dashboard /app ed elenco viaggi; con E2E_TRIP_ID anche il dettaglio di un trip reale. Comando: npm run screenshots:presentation — vedi README.md (serve .env e npx playwright install firefox).

1. Panoramica

EasyTrip risolve la frammentazione della pianificazione: invece di note sparse e blog generici, l’utente ottiene un itinerario giorno per giorno con slot (mattina / pomeriggio / sera), ristoranti, suggerimenti locali e dati geocodificati persistiti nel database come JSON strutturato (Day, TripVersion).

Problema

  • Troppo tempo su ricerca e tab aperte
  • Proposte non calibrate su durata breve (2–5+ giorni)
  • Scarsa gestione gruppo e spese sullo stesso viaggio

Soluzione (nel prodotto)

  • Generazione AI orchestrata (Inngest + Anthropic)
  • Unlock progressivo per giorno (unlockDate)
  • Carosello fino a 7 versioni per trip (MAX_TRIP_VERSION)
  • Inviti gruppo, split spese e saldi tra membri

Target (allineato alla landing)

Viaggiatori che pianificano weekend e soggiorni brevi in Europa (coppie, piccoli gruppi, early adopter beta). Copy di prodotto in src/app/page.tsx: «Itinerari AI per Weekend in Europa · Pianifica in 30 secondi».

2. Strategia di prodotto

User personas

Elena — Organizzatrice

Crea il trip, paga il checkout Stripe, avvia la generazione, invita il gruppo con invite_token, confronta le versioni e rigenera quando serve.

Marco — Membro del gruppo

Entra tramite /join/[token], consulta l’itinerario, registra spese e verifica i saldi (TripMember.balance).

Sofia — Power user / beta

Usa sostituzione slot, suggerimenti live, referral e waitlist; sensibile a qualità percepita (geo_score, rating versione).

Value proposition

  • ✓ Risparmio di tempo — da preferenze a itinerario strutturato senza copia-incolla manuale.
  • ✓ Output strutturato e persistente — giorni, slot, mappe (Leaflet) e metriche di qualità in DB.
  • ✓ Controllo e iterazione — fino a 7 versioni; rigenerazioni gratuite (v2–v4) e a pagamento (v5–v7) da trip-regen-rules.ts.
  • ✓ Monetizzazione chiara — prezzi configurabili in centesimi EUR via env (vedi modello di business).

Modello di business (implementato)

Voce Default (env) Riferimento codice
Solo / coppia €9,99 STRIPE_PRICE_SOLO_COUPLE_CENTS=999
Gruppo €14,99 STRIPE_PRICE_GROUP_CENTS=1499
Rigenerazione (v5–v7) €1,99 STRIPE_PRICE_REGEN_CENTS=199
Riattivazione post-trip €2,90 STRIPE_PRICE_REACTIVATE_CENTS=290

Tipi pagamento Prisma: purchase · regen · reactivate. Crediti e referral: modelli Credit, Referral con stati pending → signed_up → converted. Campo planType sullo user (free/sub) predispone espansione subscription.

3. Dettagli operativi

Workflow tecnico (sotto il cofano)

  1. L’utente autenticato (Clerk) crea un Trip via POST /api/trips.
  2. POST /api/billing/checkout crea sessione Stripe; al successo il webhook POST /api/webhooks/stripe aggiorna stato e pagamenti.
  3. POST /api/trips/[tripId]/generate invia evento Inngest (es. trip/generate.requested); in produzione richiede pagamento; in dev può essere forzato.
  4. La funzione generateItinerary chiama Anthropic (ANTHROPIC_MODEL, default claude-sonnet-4-20250514), valida con Zod e persiste TripVersion + Day.
  5. Email «itinerary ready» via Resend se configurato; altrimenti log lato server (transactional.ts).
  6. Il client mostra dettaglio, mappa Leaflet, carosello versioni (POST .../active-version), sostituzione slot e live suggest sulle API dedicate.
  7. Job schedulati Inngest: scadenze trip, reminder pre/post viaggio, drip waitlist, reminder scadenza crediti.

Funzionalità principali (tracciate nel codice)

  • TripService — CRUD trip, permessi organizzatore/membro
  • BillingService + BillingController — checkout / regen / reactivate
  • SlotReplaceService — replace slot (GPS / haversine dove applicabile)
  • LiveSuggestService — suggerimenti contestuali
  • ExpenseService — spese e ripartizioni
  • SupportService — ticket e messaggi
  • ReferralService — codici e conversioni
  • AuthService — sync utente Clerk ↔ DB

Metriche di successo suggerite

Metrica Formula / nota
North Star Trip con generazione completata e versione attiva (evento + record TripVersion.isActive)
Conversione checkout Sessioni Stripe completate / trip creati in stato pending → active
Attach rate rigenerazioni Pagamenti regen / utenti con regenCount ≥ 4
Qualità percepita Distribuzione geo_score e userRating per versione
Retention post-trip Tasso reactivate; crediti usati entro expiresAt
Crescita referral ReferralStatus.converted / link condivisi

4. Mercato

Il mercato dei travel planner va dai contenuti statici ai tool solo-mappa. EasyTrip si posiziona come prodotto transazionale + AI operativa: l’utente paga per un deliverable strutturato, può iterare (versioni) e gestire il viaggio in gruppo.

Area Approccio tipico competitor EasyTrip (repo)
Output Articoli o liste non strutturate Schema giorni/slot/ristoranti in DB + UI dedicata
Iterazione Rifare query da zero Fino a 7 versioni, switch attivo, regole free/paid
Monetizzazione Ads o abbonamenti generici Stripe checkout per trip + regen + reactivate
Gruppo Tool separati (chat, fogli) Invite token, membri, spese e saldi integrati
Lingua / focus Mercato globale anonimo UX e copy orientati IT / weekend Europa

Strategia di monetizzazione

Modello attuale: fee per viaggio differenziata solo/coppia vs gruppo, micro-transazioni sulle rigenerazioni premium e sulla riattivazione, crediti collegabili a referral. Roadmap plausibile: piano subscription (planType, subExpiresAt) per N trip/mese o rigenerazioni incluse — da validare con ricerca prezzo e CAC/LTV.

5. Specifiche tecniche

Product spec sintetica (MVP in codebase)

  • Auth Clerk su area /app (middleware.ts)
  • CRUD viaggi, generazione asincrona, versioni, preferenze, archiviazione soft-delete
  • Pagamenti Stripe (checkout + webhook) e tipi pagamento allineati a Prisma
  • Gruppo: invite + join token; spese con categorie enum; saldi
  • Support ticket + messaggi; waitlist + drip Inngest
  • Referral track + reward credit; analytics PostHog
  • Test E2E Playwright (tests/e2e/)

User stories (legate al codice)

  • Come organizzatore voglio pagare e avviare /api/trips/:id/generate così ricevo un itinerario salvato in TripVersion/Day.
  • Come membro voglio usare /api/join/[token] per associarmi al trip e vedere le stesse informazioni.
  • Come utente voglio cambiare versione attiva con POST /api/trips/:id/active-version entro il limite di 7 versioni.
  • Come utente voglio rigenerazioni gratuite (v2–v4) e a pagamento (v5–v7) coerenti con trip-regen-rules.ts e regen-checkout.
  • Come utente voglio sostituire uno slot via replace-slot e ottenere suggerimenti live da live-suggest.
  • Come partecipante voglio aggiungere spese e consultare GET .../balances.
  • Come utente voglio riaprire un viaggio scaduto con reactivate-checkout.
  • Come beta voglio iscrivermi in waitlist (POST /api/waitlist) e ricevere drip automatizzati.

MVP scope — checklist repository

  • ✓ API Route Handlers sotto src/app/api/
  • ✓ Prisma schema + PostgreSQL
  • ✓ Inngest: generateItinerary, expireTrips, creditExpiryReminders, waitlistDrip, preTripReminders, postTripFollowup
  • ✓ Dev: npm run inngest:dev + endpoint /api/inngest
  • ✓ E2E Playwright (signup, checkout)
  • ✓ Documentazione aggiuntiva in docs/desktop-html/
